# 编译器优化深入理解

## 1. 编译优化等级 -O0 / -O1 / -O2 / -O3

编译器优化的目的：

```text
提高程序执行效率
减少代码大小
降低内存访问
充分利用CPU硬件特性
```

GCC/Clang 常用优化等级：

```bash
-O0
-O1
-O2
-O3
-Os
-Ofast
```

---

# 2. -O0：关闭优化（Debug默认）

## 含义

```bash
gcc -O0 main.c
```

表示：

```text
不进行代码优化
```

编译器尽量保持：

```text
源码
 ↓
机器代码
```

之间的对应关系。

---

## 特点

### 优点

* 调试最容易
* 变量不会被优化掉
* 单步执行与源码一致

### 缺点

* 执行效率低
* 代码体积大
* CPU利用率低

---

## 示例

源码：

```c
int add(int a,int b)
{
    return a+b;
}
```

-O0可能生成：

```asm
push rbp

mov rbp,rsp

mov eax,[a]

add eax,[b]

pop rbp

ret
```

严格按照：

```text
参数保存
计算
返回
```

执行。

---

# 3. -O1：基础优化

开启：

```bash
gcc -O1 main.c
```

目标：

```text
减少明显的性能浪费
```

---

## 常见优化

### 删除无用代码

源码：

```c
int func()
{
    int a=10;

    return 0;
}
```

变量：

```c
a
```

没有使用。

优化：

```c
int func()
{
    return 0;
}
```

---

### 简单内联

源码：

```c
inline int add(int a,int b)
{
    return a+b;
}
```

可能展开：

```c
x = 1+2;
```

---

### 常量传播

源码：

```c
int a=10;

int b=a+5;
```

优化：

```c
int b=15;
```

---

# 4. -O2：生产环境常用

命令：

```bash
gcc -O2 main.c
```

这是 Linux Kernel、嵌入式产品常使用级别。

---

## 包含

-O1所有优化。

增加：

---

## 4.1 函数内联

原：

```c
int add(int a,int b)
{
    return a+b;
}


x=add(1,2);
```

优化：

```c
x=1+2;
```

减少：

```text
函数调用
压栈
跳转
返回
```

---

## 4.2 死代码删除

源码：

```c
if(0)
{
    send_data();
}
```

删除：

```text
send_data()
```

---

## 4.3 循环优化

源码：

```c
for(i=0;i<100;i++)
{
    sum+=a[i];
}
```

优化：

* 调整循环结构
* 减少判断
* 提高流水线效率

---

## 4.4 寄存器优化

源码：

```c
int count;

count++;
count++;
```

可能：

```asm
add r0,#2
```

减少内存访问。

---

# 5. -O3：激进优化

命令：

```bash
gcc -O3 main.c
```

目标：

```text
最大执行速度
```

---

增加：

## 5.1 更激进内联

例如：

```c
func1()
{
}

func2()
{
}
```

可能全部展开。

---

## 5.2 循环展开

原：

```c
for(i=0;i<4;i++)
{
    sum+=a[i];
}
```

变成：

```c
sum+=a[0];
sum+=a[1];
sum+=a[2];
sum+=a[3];
```

---

## 5.3 自动向量化

例如：

```c
for(i=0;i<4;i++)
{
    c[i]=a[i]+b[i];
}
```

普通：

```text
一次处理一个数据
```

优化：

ARM NEON：

```text
一次处理4个float
```

---

# 6. -O2 和 -O3 区别

| 项目   | O2 | O3   |
| ---- | -- | ---- |
| 优化程度 | 高  | 最高   |
| 代码大小 | 较小 | 可能变大 |
| 执行速度 | 快  | 可能更快 |
| 内联   | 适度 | 激进   |
| 循环展开 | 少  | 多    |
| 向量化  | 部分 | 更多   |

# 7. 为什么调试时变量会“消失”

这是嵌入式开发非常常见的问题。

例如：

```c
int test()
{
    int a=10;

    return 0;
}
```

编译：

```bash
gcc -O2 -g test.c
```

调试：

```gdb
print a
```

结果：

```text
No symbol "a"
```

为什么？

---

# 8. 原因1：变量被优化删除

编译器发现：

```text
a
```

没有任何作用。

于是：

源码：

```c
int a=10;

return 0;
```

优化：

```c
return 0;
```

变量不存在。

---

# 9. 原因2：变量进入寄存器

源码：

```c
int counter;

counter++;
```

-O0：

```text
内存
 |
counter
```

-O2：

```text
CPU寄存器
 |
r0
```

调试器查看：

```text
内存没有
```

所以找不到。

---

# 10. 原因3：变量生命周期改变

源码：

```c
{
 int a;

}

printf();
```

编译器：

```text
a作用结束
释放寄存器
```

但是源码认为：

```text
变量还存在
```

产生差异。

---

# 11. 为什么内联函数提高性能

普通函数调用：

```text
调用者
↓
保存现场
↓
跳转函数
↓
执行
↓
恢复现场
↓
返回
```

存在：

* 参数传递
* 栈操作
* 跳转开销

---

## 示例

函数：

```c
int add(int a,int b)
{
    return a+b;
}
```

调用：

```c
x=add(1,2);
```

普通：

```asm
call add

return
```

内联：

```asm
mov x,3
```

减少：

```text
call指令
return指令
栈操作
```

---

# 12. 为什么循环展开提高DSP/ARM性能

CPU执行循环：

```c
for(i=0;i<4;i++)
{
    y+=a[i];
}
```

每次：

```text
比较i
跳转
增加i
访问数据
计算
```

大量控制开销。

---

展开：

```c
y+=a[0];
y+=a[1];
y+=a[2];
y+=a[3];
```

减少：

```text
循环判断
跳转
计数更新
```

# 13. DSP/ARM中特别有效原因

## 13.1 提高流水线利用率

CPU流水：

```text
取指
译码
执行
访存
写回
```

循环跳转：

```text
流水线刷新
```

展开：

```text
连续指令流
```

效率更高。

---

## 13.2 配合SIMD

ARM NEON：

一次：

```text
4个float
```

计算。

例如：

普通：

```text
a0+b0

a1+b1

a2+b2

a3+b3
```

NEON：

```text
VADD.F32
一次完成
```

---

## 13.3 减少分支预测失败

循环：

```text
branch
branch
branch
```

展开：

```text
连续执行
```

减少预测压力。

# 14. GCC和LLVM代码质量区别

## GCC

特点：

* 历史悠久
* Linux默认
* 嵌入式大量使用

优势：

* 稳定
* 支持架构多
* ARM支持成熟

例如：

```text
ARM Cortex-A
ARM Cortex-M
PowerPC
MIPS
```

---

## LLVM/Clang

特点：

现代编译框架。

结构：

```text
C/C++
  |
Clang
  |
LLVM IR
  |
优化
  |
机器码
```

---

# 15. GCC vs LLVM

| 项目   | GCC    | LLVM    |
| ---- | ------ | ------- |
| 架构   | 传统编译器  | 模块化框架   |
| 中间表示 | GIMPLE | LLVM IR |
| 优化能力 | 强      | 强       |
| 代码生成 | 成熟     | 先进      |
| 编译速度 | 较慢     | 较快      |
| 嵌入式  | 大量使用   | 越来越多    |

# 总结

编译优化本质：

```text
减少无效工作
减少内存访问
提高CPU流水线利用率
利用硬件并行能力
```

关键关系：

```
-O0
 ↓
源码对应最好
性能最低


-O2
 ↓
工程最常用
性能和稳定平衡


-O3
 ↓
极限性能
可能增加代码体积
```

对于 ARM/DSP/基带软件：

最重要优化：

1. 函数内联
2. 循环展开
3. SIMD向量化
4. 寄存器优化

6. cache友好布局
